# HTTP/1.0, HTTP/1.1, HTTP/2
## HTTP/1.0
하나의 연결이 하나의 요청을 처리하도록 설계되었다. 요청마다 TCP 연결을 새로 맺고 끊어야 했기 때문에 통신 속도가 느리다는 문제가 있었다.

## HTTP/1.1
### Keep-Alive
지정된 시간 동안 TCP 연결을 끊지 않고 유지한다. 하나의 연결을 통해 여러 개의 요청/응답을 처리할 수 있게 되면서 통신 속도가 획기적으로 개선되었다.

### Pipelining
클라이언트는 앞 요청의 응답을 기다리지 않고 여러 개의 요청을 보낼 수 있다. TCP 특성상 요청 후 응답을 기다리는 문제를 보완하기 위해 추가되었다.

하지만, 파이프라이닝은 서버에서 응답을 반드시 순서대로 보내야 한다. 앞의 요청에 대한 처리가 늦어지면 뒤의 모든 요청이 전송되지 못하고 대기해야 한다는 문제가 있었다.
이를 HOL Blocking(Head Of Line Blocking)이라고 부른다.

## HTTP/2.0
### Frame
이전까지는 메시지를 text로 보냈지만, HTTP/2.0은 메시지를 이진화하여 frame이라는 데이터 단위를 만든다.
1개의 메시지는 1~2개의 HEADER frame과 0개 이상의 DATA frame으로 나뉘어진다.

### Multiplexed Streams(멀티플렉싱)
하나의 연결 내에서 여러 개의 스트림을 가질 수 있게 되어 다중화(multiplexing)가 가능해졌다.
스트림을 통해 각 요청의 전송 순서가 의미가 없어지면서, 애플리케이션 계층의 HOL Blocking 문제를 해결했다.

하지만, 여전히 TCP의 작동 방식으로 인해 발생하는 전송 계층의 HOL Blocking 문제는 해결하지는 못했다.
패킷 유실이 발생하는 경우 TCP 계층은 유실된 패킷이 재전송 되어 도착할 때까지 패킷들을 애플리케이션 계층으로 올려보내지 않는다.
이 때문에 서로 다른 스트림이 전송되고 있을 때, 하나의 스트림에서 유실이 발생하는 경우 다른 스트림까지 지연되는 현상이 발생한다.

HTTP/2.0은 기본적으로 하나의 TCP 커넥션을 계속 유지하면서 멀티플렉싱으로 여러 요청을 동시에 처리하는 구조이기 때문에 keep-alive 헤더는 deprecated 되었다.

### Header Compression
요청과 응답 헤더의 메타데이터를 압축해서 기존의 연속된 요청에서 중복 헤더로 인한 오버헤드 문제를 해결했다.

### Server Push
클라이언트가 서버에 요청하지 않아도 클라이언트에게 필요할 것으로 예상되는 리소스를 추가로 전송할 수 있는 기능이 추가되었다.
하지만, 리소스 낭비 문제로 인해 현재는 주요 브라우저에서 지원이 중단되었다.
